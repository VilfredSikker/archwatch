Form Factor Decision

Architecture diagrams need visual space for layout, zoom, hover tooltips, and animation. The form factor determines what's possible.

🌐

Web App (Local Server + Browser)

★ RECOMMENDED

PROS

  • +Full SVG/Canvas rendering — real graph layouts, zoom, pan, animations
  • +Zoomable detail levels (module → file → function) work naturally
  • +WebSocket for live push updates — sub-100ms change reflection
  • +Hover tooltips, click-to-navigate, minimap, search all come free
  • +CSS animations for change highlighting (pulses, glows, trails)
  • +Can screenshot/export diagrams easily
  • +Works alongside terminal — er in one pane, diagram in browser

CONS

  • −Requires a local server process (Rust binary serves HTTP + WebSocket)
  • −Browser tab management — yet another window to arrange
  • −Slightly more complex deployment (though still single binary)
  • −Frontend code adds JS/HTML to maintain alongside Rust
⬛

TUI (Terminal App)

PROS

  • +Consistent with ER aesthetic — same dark terminal vibe
  • +Zero dependencies, single binary, no browser needed
  • +Can reuse ratatui + crossterm + notify from ER if embedded
  • +Keyboard-driven navigation matches developer workflow

CONS

  • −Architecture diagrams in ASCII are fundamentally limited — can't show complex graph layouts, overlapping edges, or smooth zoom
  • −Zoomable module→function drill-down is very hard in box-drawing
  • −No hover tooltips, no minimap, limited color palette
  • −Change animations limited to color flashes (no smooth transitions)
  • −Terminal width constrains layout — 80-col repos would be cramped
  • −If embedded in ER, adds significant complexity to an already 19K-line codebase
🔀

Hybrid (TUI + Browser)

PROS

  • +TUI for quick glance + keyboard nav, browser for full exploration
  • +Degrades gracefully (SSH = TUI only, local = full browser)

CONS

  • −Two UIs to build and maintain — doubles the work
  • −TUI version will always feel like a lesser experience
  • −Adds complexity without proportional value
  • −Users will just use the browser version and ignore TUI
📦

Embedded in Easy Review

PROS

  • +Reuses file watcher, diff parsing, git integration
  • +Diagram context right next to the diff you're reviewing
  • +Single tool to learn and run

CONS

  • −ER is already 19K lines with complex state — adding graph layout would bloat it
  • −Same TUI rendering limitations (ASCII diagrams)
  • −Different use case: ER is for reviewing diffs, this is for understanding architecture
  • −Screen real estate conflict — diff view needs full width
  • −Code analysis (imports, call graph) is unrelated to diff parsing

THE VERDICT: STANDALONE WEB APP

Architecture diagrams are fundamentally visual. They need graph layout algorithms (force-directed, hierarchical), zoom/pan, hover tooltips, animated transitions, and space to breathe. Terminal box-drawing can't deliver this — it would feel like viewing Google Maps in ASCII.

Keep it separate from ER. ER reviews diffs. This tool visualizes architecture. Different jobs, different interfaces. They complement each other: run er in your terminal to review changes, open aw in your browser to see where those changes fit in the system. They can share the same file watcher infrastructure via a shared Rust library crate.

Single binary, still. aw compiles to one binary. Run it, it starts a local server, opens your browser. Frontend is embedded in the binary (rust-embed). No npm, no build step for end users.

AW INTERNAL ARCHITECTURE

aw (binary)
├── analyzer/           Code analysis engine
│   ├── tree-sitter       Parse imports, functions, structs, traits
│   ├── module_graph      Build module → file → symbol dependency graph
│   ├── change_detector   Map git diffs to graph nodes
│   └── cache             Incremental analysis (don't re-parse unchanged files)
│
├── layout/              Graph layout algorithms
│   ├── force_directed    Physics sim for organic layouts (d3-force in Rust)
│   ├── hierarchical      Top-down layer assignment (Sugiyama-style)
│   ├── zoom_levels       Module → File → Symbol detail switching
│   └── animation         Smooth transitions between states
│
├── watcher/             File system monitoring
│   ├── notify + debounce Shared with ER (extract to crate)
│   ├── git_events        Detect commits, staging, branch switches
│   └── diff_mapper       Map changed files → affected graph nodes
│
├── server/              HTTP + WebSocket server
│   ├── axum              Lightweight async HTTP
│   ├── websocket         Push updates to browser in real-time
│   ├── rust-embed        HTML/JS/CSS baked into binary
│   └── api               REST endpoints for diagram data
│
└── frontend/            Embedded web UI (served from binary)
    ├── d3.js             SVG rendering + force layout
    ├── websocket client  Receive live updates
    ├── zoom/pan          Smooth navigation between detail levels
    └── change_highlight  Pulse, glow, trail animations

BUILD PHASES

PHASE 1

Static Diagram Generation — 1-2 weeks

Goal: aw scans a repo and opens a browser with an interactive architecture diagram.

Parser: Use tree-sitter (Rust bindings) to extract imports, module structure, and public symbols. Start with Rust since ER is the test case.

Layout: D3 force-directed graph in the browser. Nodes = modules/files, edges = import dependencies. Cluster by directory.

Server: Axum serves the embedded frontend + a /api/graph endpoint returning JSON.

Output: aw → opens localhost:3838 → interactive SVG diagram of the repo.

PHASE 2

Live Watching + Change Highlighting — 1-2 weeks

File watcher: Extract ER's notify watcher into a shared crate. Watch for file saves.

WebSocket: On file change → re-run tree-sitter on changed files only (incremental) → compute affected nodes → push delta to browser.

Animations: Changed nodes pulse green. Recently changed nodes glow yellow. Edges to changed nodes highlight. Change fades over 30 seconds.

Activity feed: Left sidebar shows chronological change log with timestamps.

PHASE 3

Zoomable Detail Levels — 1-2 weeks

Module level: Directories as clustered groups. Click to expand.

File level: Individual files visible with line counts and key exports.

Symbol level: Functions, structs, traits, enums inside a file. Call graph edges.

Semantic zoom: Pinch/scroll zoom smoothly transitions between levels via D3 zoom thresholds.

PHASE 4

Polish + ER Integration — 1 week

ER link: er gets a hotkey (a) to open aw for the current repo. Shares base branch detection.

Heatmap mode: Color nodes by change frequency over time (git log analysis).

Export: SVG/PNG export of current diagram state. Useful for docs and PRs.

Multi-language: Extend tree-sitter grammars for TypeScript, Python, Go.

KEY TECHNICAL DECISIONS

tree-sitter vs regex for code analysis?

Tree-sitter. It gives us a real AST — we can find imports, function signatures, struct definitions, trait impls. Regex would break on edge cases and can't give us the symbol-level zoom. tree-sitter has Rust bindings and grammars for every language we care about.

D3 in the browser vs Rust-side layout?

D3 in browser. Force-directed layout needs fast iteration and smooth animation — JavaScript excels here. Rust backend provides the graph data, browser handles layout + rendering.

axum vs warp vs actix for the server?

Axum. Lightweight, tokio-native, good WebSocket support. We only need 3 endpoints: serve embedded frontend, REST for graph data, WebSocket for live updates.

Shared crate with ER?

Extract a er-common crate with file watching, git helpers, and base branch detection. Both er and aw depend on it. Start with copy-paste in Phase 1, extract in Phase 4.

Incremental analysis?

Cache the parsed AST per file, keyed by content hash. On file change, only re-parse that file and update its edges. Full repo re-scan only on branch switch. Keeps watch-mode updates under 50ms.

TOTAL ESTIMATE: 4-7 WEEKS

Phase 1 is the foundation and delivers value immediately — you can see your repo's architecture. Each subsequent phase adds a layer of polish. The tool is useful from day one and gets better incrementally.
